# Paladin 開発ロードマップ

## 1. 目的

本ロードマップは、Paladin を一気に作り込むのではなく、小さく安定した単位で段階的に育てるための実装計画を示すものである。

Paladin の初期価値は、多機能さではなく、生成 AI や人間が安定して扱える「設計ルール診断器」を成立させることにある。
そのため、最初期は `check` コマンドを中心に据え、構造化された診断結果を返す基盤を先に安定させる。

## 2. 全体方針

本ロードマップは、次の方針で進める。

- 最初は `check` コマンドに集中する
- AST 解析基盤より先に、診断結果の契約を最小限でも固定する
- 最初のルールは 1 つだけ実装し、ルール追加可能な構造を先に作る
- 出力は人間向け表示を持ちつつ、内部的には構造化診断を前提にする
- Ignore、設定ファイル、自動修正、IDE 連携は後段へ回す
- 初期段階では単一ファイル内で判定可能なルールを対象にする

## 3. フェーズ構成の概要

本ロードマップは、以下の順で進める。

1. 仕様固定フェーズ
2. 対象列挙フェーズ
3. AST 生成フェーズ
4. ルール実行基盤フェーズ
5. MVP 完成フェーズ
6. ルール拡張フェーズ
7. JSON 正式対応フェーズ
8. 運用補助コマンド拡張フェーズ
9. Ignore / 設定ファイルフェーズ
10. 複数ファイル解析フェーズ
11. 修正支援フェーズ

## 4. フェーズ詳細

## Phase 0. 仕様固定

### 目的

AST 解析やルール実装に入る前に、Paladin が何を返すツールなのかを先に固定する。
これにより、後続の実装が「診断器としての契約」を中心に揃うようにする。

### このフェーズで決めること

#### 0-1. `check` の実行結果状態

少なくとも次の 3 状態を固定する。

- 違反なし
- 違反あり
- 実行失敗

#### 0-2. 終了コード

終了コードは次の固定仕様とする。

```text
0 = 違反なし
1 = 違反あり
2 = 実行失敗
````

初期段階では、この仕様をオプションで切り替え可能にはしない。

#### 0-3. 最小データモデル

内部的に、少なくとも次の概念を定義する。

- `Violation`
- `CheckSummary`
- `CheckResult`

想定する責務は次の通り。

##### `Violation`

各違反を表現する。

少なくとも以下を持つ。

- `file`
- `line`
- `column`
- `rule_id`
- `rule_name`
- `message`
- `reason`
- `suggestion`

##### `CheckSummary`

実行全体の集計情報を表現する。

少なくとも以下を持つ。

- `total_violations`
- `violations_by_rule`
- `violations_by_file`
- `checked_files`
- `files_with_violations`

##### `CheckResult`

実行結果全体を表現する。

少なくとも以下を持つ。

- `status`
- `summary`
- `diagnostics`

#### 0-4. 出力契約

初期実装では `text` 出力のみでもよいが、内部モデルは JSON に自然に変換できる前提で設計する。

### 完了条件

- `check` の 3 状態が定義されている
- 終了コード仕様が固定されている
- `Violation / CheckSummary / CheckResult` の最小モデルが定義されている
- 出力仕様の骨格が文書化されている

### このフェーズで実装しないこと

- 実際の AST 解析
- 実際のルール判定
- 設定ファイル
- Ignore
- 自動修正

### 後回し理由

初期に最も重要なのは、Paladin が「何を返すツールか」を安定させることである。
ここを曖昧にしたまま AST やルール実装へ進むと、後から出力契約や終了コードを変更する必要が生じ、設計が崩れやすい。

## Phase 1. 対象列挙

### 目的

指定されたファイルやディレクトリから、解析対象となる Python ファイル群を安定して取得できるようにする。

### 実装内容

- ファイルパスまたはディレクトリパスを受け取る
- ディレクトリ配下を再帰的に探索する
- `.py` ファイルのみを対象とする
- 重複を排除する
- 順序を安定化させる

### 完了条件

- `uv run paladin check src/` のような入力から `.py` 一覧を取得できる
- 単一ファイルと複数ディレクトリの両方を扱える
- 同一入力に対して、同じ順序で対象ファイルが得られる

### このフェーズで実装しないこと

- AST 生成
- ルール判定
- Ignore パターン
- 設定ファイルによる除外

### 後回し理由

まずは最小限の対象列挙を安定させるべきであり、除外設定や柔軟な対象制御は後から追加できる。
初期に対象制御を複雑にすると、列挙基盤そのものの責務が曖昧になる。

## Phase 2. AST 生成

### 目的

列挙した各 Python ファイルから、ファイル単位で AST を構築できるようにする。

### 実装内容

- 各ファイルを読み込む
- ファイル単位で AST を生成する
- AST と元ファイルパスを対応づける
- 読み込み失敗や構文解析失敗を実行失敗として扱う

### 補足方針

初期段階では、構文エラーを「ルール違反」として扱わず、Paladin 自体の実行失敗として扱う。
将来的に構文エラーも診断対象へ含める拡張は可能だが、初期責務は設計ルール診断に限定する。

### 完了条件

- 対象ファイルごとに AST が生成できる
- AST 生成失敗時に `status=error` 相当へ落とし込める
- AST とファイルパスの対応が保持されている

### このフェーズで実装しないこと

- ルールごとの AST 走査
- 複数ファイルをまたいだ解析
- 構文エラーの診断出力化

### 後回し理由

AST を作る段階では、まだ「何を違反とみなすか」を持ち込まないほうがよい。
解析の入口と診断ロジックを分離することで、後からルールを追加しやすくする。

## Phase 3. ルール実行基盤

### 目的

AST に対してルール群を適用し、`Violation` を収集する共通基盤を作る。

### 実装内容

#### 3-1. ルールの最小インターフェイス定義

少なくとも次の責務を分離する。

- `Rule`
    - 単一ルールのみを判定する
- `RuleRunner` または `RuleSet`
    - 複数ルールを束ねて実行する
- `RuleRegistry`
    - 利用可能ルールの一覧とメタ情報を管理する

#### 3-2. 最初のルールを 1 本だけ実装

最初のルールは、次の条件を満たすものを優先する。

- 単一ファイル内で完結する
- AST のみで判定できる
- 違反箇所の位置を返しやすい
- 誤検出しにくい
- 修正の方向性を書きやすい

#### 3-3. `Violation` 生成

ルール違反時には、`Violation` を 1 件ずつ追加する。
無関係な問題を 1 件にまとめすぎない。

### 完了条件

- ルール 1 本が AST を入力にして違反を返せる
- 複数ファイルへ同じルールを適用できる
- `RuleRunner` が `Violation` 一覧を収集できる
- ルール追加時に既存コードの変更範囲が限定される構造になっている

### このフェーズで実装しないこと

- 多数のルール追加
- ルールごとの設定項目
- Ignore コメント
- 自動修正

### 後回し理由

ここで重要なのは、ルール数ではなく「ルールを継続的に増やせる骨格」を先に作ることである。
最初から多くのルールを入れると、設計の不備がルール間に広がりやすい。

## Phase 4. MVP 完成

### 目的

`check` コマンドを、実用可能な最小の診断器として成立させる。

### 実装内容

- `Violation` 一覧を人間向けテキストとして標準出力する
- `CheckSummary` を表示する
- 終了コードを返す
- 違反あり・違反なし・実行失敗を機械的に判別可能にする

### 期待する出力イメージ

最低限、次を表現できること。

- 各違反の対象ファイル
- 行番号
- ルール識別子
- 違反概要
- 違反理由
- 修正の方向性
- 総違反件数
- ルール別違反件数
- ファイル別違反件数
- 実行結果の要約

### 完了条件

- `uv run paladin check src/` が end-to-end で動く
- 違反が 1 件以上あれば終了コード 1 を返す
- 違反がなければ終了コード 0 を返す
- 実行失敗時は終了コード 2 を返す
- 診断結果が読み取り可能なテキストとして出力される

### このフェーズで実装しないこと

- JSON 出力の正式公開
- `rules` コマンド
- `version` コマンド
- Ignore
- 設定ファイル

### 後回し理由

MVP の成立条件は「1 本のルールで安定して違反を検出し、固定契約で返せること」である。
補助コマンドや柔軟な設定は、その後でも十分追加可能である。

## Phase 5. ルール拡張

### 目的

ルール基盤が安定していることを確認しながら、ルール資産を増やす。

### 実装内容

- 単一ファイル内で完結するルールを追加する
- ルール追加時の実装パターンを定型化する
- ルールメタ情報の記述方式を整える

### 優先順

次の順で増やす。

1. AST のみで判定できる単純ルール
2. 位置情報が取りやすいルール
3. 修正の方向性を書きやすいルール
4. 設定なしで使えるルール

### 完了条件

- ルールを複数本追加しても基盤が崩れない
- 新規ルール追加の実装手順が明確化されている
- ルールごとの差分がロジック中心で、基盤改修が最小限で済む

### このフェーズで実装しないこと

- ルール個別設定
- ディレクトリ別ルール制御
- 複数ファイル解析

### 後回し理由

まずは単一ファイル診断の品質と保守性を高めるべきであり、設定や広域解析に進むのはその後でよい。

## Phase 6. JSON 正式対応

### 目的

Paladin の診断結果を、生成 AI や他ツールが確実に再利用できるようにする。

### 実装内容

- `--format json` を正式対応する
- `CheckResult` を JSON シリアライズ可能にする
- `text` と `json` で意味が一致するようにする

### 補足方針

初期から内部は構造化前提で作るが、ここで初めて JSON を対外的な正式機能として扱う。
必要であれば `ndjson` はこの次でもよい。

### 完了条件

- `uv run paladin check src/ --format json` が使える
- `status / summary / diagnostics` が安定して出力される
- 同一入力に対して JSON の構造が安定している

### このフェーズで実装しないこと

- Markdown レポート出力
- HTML レポート出力
- 自動修正用パッチ出力

### 後回し理由

まず重要なのは、機械処理しやすい JSON を安定させることである。
Markdown や装飾的なレポートは補助価値であり、優先度は下がる。

## Phase 7. 運用補助コマンド拡張

### 目的

ルールを継続的に育てやすいよう、補助的な CLI を追加する。

### 実装候補

- `rules`
- `version`
- 必要に応じて `explain`

### 優先順

1. `rules`
2. `version`
3. `explain`

### 完了条件

- 利用可能ルール一覧を確認できる
- 実行している Paladin のバージョンを取得できる
- 必要に応じて個別ルールの説明を参照できる

### このフェーズで実装しないこと

- `fix`
- `serve`
- `watch`
- `lsp`

### 後回し理由

これらは有用だが、診断器としての価値成立には必須ではない。
まずはルールの可視化と運用性向上に留める。

## Phase 8. Ignore 機能

### 目的

誤検出や例外的事情に対応するため、違反の抑制機構を導入する。

### 実装順の推奨

1. ファイル単位の Ignore
2. 直前コメントによる Ignore
3. 設定ファイルによる全体 Ignore
4. ディレクトリ単位の例外

### 完了条件

- 特定ルールの違反を限定的に無視できる
- Ignore の適用範囲が明確である
- Ignore によって診断結果の追跡性が失われない

### このフェーズで実装しないこと

- 高度な条件式ベース Ignore
- 実行時動的ルール評価
- ベースライン機能

### 後回し理由

Ignore は実運用で必要になりやすい一方、初期段階で入れるとルール診断基盤そのものの品質評価を難しくする。
まずは素の検出品質を見極めるべきである。

## Phase 9. 設定ファイル

### 目的

プロジェクトごとの差異を吸収しつつ、Paladin を再利用しやすくする。

### 実装順の推奨

1. ルール ON/OFF
2. 解析対象パスの include/exclude
3. ルール個別設定
4. ディレクトリ別設定

### 完了条件

- プロジェクトごとに有効ルールを制御できる
- 解析対象から除外したいパスを設定できる
- 一部ルールの挙動を設定値で調整できる

### このフェーズで実装しないこと

- 複雑な継承設定
- 環境別多段マージ
- ルールごとの任意スクリプト注入

### 後回し理由

設定ファイルは便利だが、初期に入れると「基盤が未成熟なまま設定だけ増える」状態になりやすい。
まずは設定なしでも使えるルール基盤を作ることが先である。

## Phase 10. 複数ファイル解析

### 目的

単一ファイルでは判定できない設計ルールへ拡張する。

### 対象例

- 複数ファイルにまたがるルール
- モジュール境界に関するルール
- 依存方向に関するルール
- パッケージ単位の構成ルール

### 実装内容

- import 関係の収集
- ファイル間参照モデルの構築
- パッケージ単位の判定基盤
- 広域解析時のエラーハンドリング設計

### 完了条件

- 複数ファイルにまたがるルールを少なくとも 1 本実装できる
- 単一ファイルルールと複数ファイルルールが同居できる
- 解析時間と責務の増加に耐えられる構造になっている

### 後回し理由

この領域は価値が大きい一方で、設計難易度も大きく上がる。
単一ファイル診断の骨格が安定する前に着手すると、基盤全体が複雑化しやすい。

## Phase 11. 修正支援

### 目的

違反解消を支援するが、Paladin 自体の主戦場を「問題発見」に置き続ける。

### 実装候補

- `suggestion` の改善
- AI 向け修正指示文の整備
- 一部ルールの限定的自動修正
- 将来的な `fix` コマンドの検討

### 推奨方針

自動修正より先に、生成 AI が扱いやすい修正指示の品質を高める。
自動修正は、意味保存が明確で安全なルールに限って限定導入する。

### 完了条件

- 各診断の修正方向性が十分に具体化されている
- 生成 AI が診断結果を読んで修正作業へ移りやすい
- 必要に応じて安全な自動修正を限定導入できる

### 後回し理由

Paladin の主要価値は問題発見にあり、自動修正は必須ではない。
先に診断品質と出力品質を高めたほうが、全体として価値が高い。

## 6. 初期リリースに含める範囲

初期リリースでは、少なくとも次を含める。

- `check`
- 1 本以上の単一ファイル向けルール
- 構造化前提の内部モデル
- text 出力
- 固定終了コード

余力があれば、次も初期寄りで入れてよい。

- `rules`
- `version`
- `--format json`

## 7. 初期リリースで入れないもの

初期リリースでは、次を前面に出さない。

- `fix`
- `serve`
- `lsp`
- `watch`
- `baseline`
- `ignore`
- `config validate`
- 複数ファイル解析ルール
- IDE 統合

理由は、初期価値が「安定した診断器」を成立させることにあり、周辺機能を先に広げると設計が不安定になりやすいためである。

## 8. 最初の実装順

最初のスプリントでは、次の順で進める。

1. `check` の結果契約を固定する
2. `Violation / CheckSummary / CheckResult` を定義する
3. 終了コード仕様を実装する
4. Python ファイル列挙を実装する
5. AST 生成を実装する
6. `Rule` と `RuleRunner` を実装する
7. 最初のルールを 1 本実装する
8. text 出力を実装する
9. end-to-end で `check` を成立させる
