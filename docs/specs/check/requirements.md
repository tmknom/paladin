# check モジュール要件定義

## 概要

### 目的

Python ソースコードを静的解析し、プロジェクトで定義された設計ルールへの違反を検出する。検出結果を生成 AI や人間が修正判断に利用できる診断情報として提供する。

### 解決する課題

- 設計ルール違反をコードレビューに頼らず自動検出できない
- 違反の有無を機械的に判別できる形式で出力できない
- 設計ルールを追加・変更するたびに解析基盤全体を修正しなければならない

### コンポーネントの概要

check モジュールは、ファイル列挙・AST 解析・ルール適用という3段階の処理を順に実行し、違反の一覧と実行結果を返す。解析対象はファイルまたはディレクトリ単位で指定でき、Python ファイルが再帰的に収集される。

## 機能要件

### 解析対象の収集

- ファイルまたはディレクトリを解析対象として指定できる
- ディレクトリが指定された場合、配下の Python ファイルを再帰的に収集できる
- ファイルが指定された場合、Python ファイルのみを対象に含め、それ以外は無視できる
- 複数のファイル・ディレクトリを同時に指定できる
- 重複したファイルは排除し、一定の順序で処理できる
- 存在しないパスが指定された場合、即座にエラーとして報告できる

### 構文解析

- 収集した Python ファイルを順次読み込み、AST（抽象構文木）を生成できる
- ファイルの読み込みエラーや構文エラーが発生した場合、即座にエラーとして報告できる

### ルール適用

- 登録されたすべてのルールを、解析対象の各ファイルに適用できる
- 各ルールは一意な識別子と名称を持ち、どのルールによる違反かを識別できる
- 違反が検出された場合、ファイル・行番号・列番号・ルール識別子・違反概要・違反理由・修正の方向性を含む診断情報を生成できる
- 違反がない場合は空の違反一覧を返せる

### ファイル単位の Ignore（コメントによる指定）

- ファイル先頭に `# paladin: ignore-file` コメントを記述することで、そのファイルの全ルール違反を無視できる
- ファイル先頭に `# paladin: ignore-file[rule-id]` コメントを記述することで、特定ルールの違反のみを無視できる
- 角括弧内に複数のルール ID をカンマ区切りで列挙できる（例: `# paladin: ignore-file[rule-a, rule-b]`）
- ディレクティブは shebang 行・エンコーディング宣言・通常コメント・空行・docstring の後に配置できる
- ディレクティブは最初の import 文や実行コードよりも前に配置しなければならない（それ以降は無視される）
- ディレクティブが存在しないファイルは、全ルールが通常通り適用される

### 行単位の Ignore（コメントによる指定）

- `# paladin: ignore` コメントの直後の行に対して、全ルール違反を無視できる
- `# paladin: ignore[rule-id]` コメントの直後の行に対して、特定ルールの違反のみを無視できる
- 角括弧内に複数のルール ID をカンマ区切りで列挙できる（例: `# paladin: ignore[rule-a, rule-b]`）
- ディレクティブの直後が空行の場合は、ディレクティブとして認識されない

### 設定ファイルによる Ignore

- `pyproject.toml` の `[tool.paladin.per-file-ignores]` セクションで、glob パターンに合致するファイルの特定ルール違反を無視できる
- パターン値に `["*"]` を指定することで、対象ファイルの全ルール違反を無視できる
- 同一ファイルに複数のパターンが合致する場合、すべてのパターンのルール設定を統合して適用できる
- コメントによる Ignore と設定ファイルによる Ignore は同時に適用でき、どちらかの条件を満たせば無視される

### CLI によるルール除外

- `--ignore-rule <rule-id>` オプションで、実行時に特定ルールをグローバルに無視できる
- 同オプションを複数回指定することで、複数ルールを同時に除外できる

### 設定ファイルによるルール ON/OFF

- `pyproject.toml` の `[tool.paladin.rules]` セクションで、ルールを識別子ごとに有効/無効にできる
- 無効化（`false`）に設定されたルールは解析対象から除外される
- 有効化（`true`）に設定されたルールは通常通り適用される（デフォルト動作と同じ）
- 未知のルール ID が指定された場合は警告を出力し、無視される

### 解析対象パスの設定

- `context.exclude` の glob パターンに合致するファイルを解析対象から除外できる
- 末尾スラッシュ付きパターンや拡張子なしの単純名は、ディレクトリとして扱い配下のすべてのファイルを除外できる

### 実行結果の提供

- 解析対象ファイル一覧・AST 解析結果・違反一覧を一括して返せる
- 違反の件数を取得できる
- 各違反をイテレーションして個別に参照できる

### 出力形式の切り替え

- `text`（デフォルト）と `json` の2種類の出力形式をサポートする
- JSON 形式では `status`・`summary`・`diagnostics` の3セクションを含む valid JSON を出力する
- `summary` には `total_violations`・`by_rule`・`by_file` が含まれる
- `diagnostics` の各要素には `file`・`line`・`column`・`rule_id`・`rule_name`・`message`・`reason`・`suggestion` の8フィールドが含まれる
- 同一入力に対して JSON の構造が安定していること（実行ごとに変動しない）
- 違反なしの場合は終了コード 0、違反ありの場合は終了コード 1 を返す（出力形式によらず共通）

### ルールの管理

- 登録されたルールのメタ情報（識別子・名称・概要）一覧を提供できる

## 品質要件

### 検出の安定性と再現性

同一の入力・同一のルール集合・同一の実行条件に対して、常に同一の診断結果を返すこと。診断の順序や内容が実行ごとに変動しないこと。

### Fail Fast による明確なエラー報告

存在しないパスの指定、ファイル読み込みエラー、構文エラーのいずれも、発生時点で即座にエラーとして報告すること。エラーを握りつぶして処理を継続しないこと。

### ルールの拡張性

新しいルールを既存の解析基盤に影響を与えずに追加できること。ルール実装が満たすべき契約（インターフェース）が明確に定義されていること。

## 前提条件

- Python の標準ライブラリである `ast` モジュールを使用して AST を生成する
- 解析対象は Python ファイル（`.py`）のみとする
