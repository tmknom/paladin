# view モジュール要件定義

## 概要

### 目的

`paladin view <RULE_ID>` コマンドで、指定したルールの詳細情報をユーザーに提示する。ルールの意図・見方・修正方向を一度のコマンドで確認できるようにすることで、違反への対処を支援する。

### 解決する課題

- ルール ID だけでは、そのルールが何を意図しているか把握できない
- 違反を修正する際に、ルールの guidance や suggestion をドキュメントで都度探す手間がかかる
- `paladin check` の出力結果からルール詳細へ素早く移動できる手段がない

### コンポーネントの概要

`paladin view <RULE_ID>` コマンドを実行すると、指定したルールの詳細情報を出力する。`--format` オプションで `text`（デフォルト）と `json` の出力形式を切り替えられる。

## 機能要件

### ルール詳細の表示

指定した RULE_ID に対応するルールの詳細情報を表示できる。表示される情報は以下の通り。

必須項目（全ルールで表示される）。

- Rule ID（ルール識別子）
- Name（ルール名）
- Summary（概要説明）
- Intent（設計意図）
- Guidance（違反の見方）
- Suggestion（修正方向）

任意項目（ルールに設定されている場合のみ表示される）。

- 背景と意図（`background`）: ルールの背景と設計意図の詳細説明
- 改善手順（`steps`）: 段階的な改善ステップのリスト
- 設定例（`config_example`）: 設定ファイルの記述例
- 検出パターン（`detection_example`）: ルールが検出する対象の具体例

### 出力形式の切り替え

- `--format` オプションで `text`（デフォルト）と `json` の2種類の出力形式を切り替えられる

### テキスト出力形式

- 詳細情報は各項目をラベル付きで表示し、ラベルと値が揃えられた形式で出力できる
- ラベル列の幅は最も長いラベルに合わせて自動調整される
- 任意項目はルールに設定されている場合のみ、セクション見出しと内容を追記して出力できる

### JSON 出力形式

- `rule_id`・`rule_name`・`summary`・`intent`・`guidance`・`suggestion` の 6 フィールドを含む JSON オブジェクトを出力できる
- 任意項目（`background`・`steps`・`config_example`・`detection_example`）はルールに設定されている場合のみ JSON フィールドとして含まれる

### 存在しないルール ID への対応

- 指定した RULE_ID がツールに登録されていない場合は、対象 RULE_ID を含むエラーメッセージを表示できる
- テキスト形式では `Error: Rule '<RULE_ID>' not found.` を出力する
- JSON 形式では `{"error": "Error: Rule '<RULE_ID>' not found."}` を出力する

### 終了コード

- 正常終了（ルール存在・不存在を問わず）の場合は終了コード `0` を返す
- エラー（不正な引数、予期しない例外）の場合は終了コード `2` を返す

## 品質要件

### 出力の一貫性

同一の RULE_ID に対して、常に同一の出力を返す。実行のたびに内容や整列が変動しない。

### 明確なエラー報告

不正な引数や予期しないエラーが発生した場合、即座にエラーとして報告すること。エラーを握りつぶして処理を継続しないこと。

### テスト容易性

ルール詳細の取得ロジックと出力フォーマットの生成ロジックは独立してテストできる。実際のルール登録状態に依存せずに出力形式を検証できる。

## 前提条件

### 技術基盤

- Python 3.11 以上
- paladin パッケージの rule モジュールにルールが登録されていること
- 各ルールが Rule ID・Name・Summary・Intent・Guidance・Suggestion の必須メタ情報を持つこと
- 各ルールは任意で Background・Steps・Config Example・Detection Example のメタ情報を持てること
