# rule 要件定義

## 概要

### 目的

`rule` モジュールは、Python コードに対して静的解析ルールを定義・実行するためのドメイン層を提供する。
各ルールはソースコードを検査し、規約違反を検出して違反情報を返す。
このモジュールは `check` モジュールなどの上位モジュールから利用される。

### 解決する課題

- Python プロジェクトのインポート規約・パッケージ境界・コード品質に関する違反を自動検出したい
- ルールの追加・削除を既存コードへの影響を最小化しながら行いたい
- ルールに関するメタ情報（説明・意図・改善提案）を一元管理して、ユーザーへの診断メッセージを統一したい
- ファイル単位またはプロジェクト全体を対象とした複数種類のルール判定をまとめて実行したい

### コンポーネントの概要

`rule` モジュールは、検査対象のソースファイル群を入力として受け取り、登録されたルールを一括適用し、違反の集合を出力する。
主要な役割として、ルールの一括実行管理、デフォルトルール一式の生成、各ルールが準拠する共通インターフェースの定義、
および検査で使用するソースファイル・違反・ルールメタ情報などの値オブジェクトの提供がある。

## 機能要件

### ルール実行

- 複数のルールをまとめて管理し、ソースファイル群に対して一括実行できる
- ルール単位で無効化（スキップ）できる
- ファイルパスを指定して特定ファイルに対するルールを無効化できる（per-file 無効化）

### 実装済みルール

以下のルールが提供される。

#### インポート規約

- `no-relative-import`: 相対インポート（`from . import ...`）の使用を禁止する
- `no-local-import`: 関数・メソッド・クラス内のローカルインポートを禁止する（`TYPE_CHECKING` ブロックを除外）
- `no-direct-internal-import`: 他パッケージの内部モジュールへの直接インポートを禁止する（パッケージの `__init__.py` を経由しないインポートを検出）
- `require-qualified-third-party`: サードパーティライブラリの `from X import Y` 形式およびエイリアス（`import X as Y`）インポートを禁止する

#### パッケージ公開インターフェース

- `require-all-export`: `__init__.py` に `__all__` の定義を要求する
- `no-non-init-all`: `__init__.py` 以外のファイルへの `__all__` 定義を禁止する
- `no-cross-package-reexport`: `__init__.py` の `__all__` に別パッケージのシンボルを再エクスポートすることを禁止する
- `no-unused-export`: `__init__.py` の `__all__` に定義されたシンボルが別パッケージから利用されていない場合に報告する

#### パッケージ境界

- `no-cross-package-import`: `allow-dirs` に含まれないパッケージからのクロスパッケージインポートを禁止する（エントリーポイント `main()` 関数を持つファイルを除外）
- `no-third-party-import`: `allow-dirs` に含まれないディレクトリからのサードパーティライブラリインポートを禁止する

#### コード品質

- `no-mock-usage`: `unittest.mock` の `Mock` / `MagicMock` のインポートを禁止する
- `no-deep-nesting`: 関数・メソッド内のネスト深度が 3 段階以上になることを禁止する
- `max-method-length`: メソッド/関数の行数が上限を超えた場合に報告する（docstring を除外した行数で計算）
- `max-class-length`: クラスの行数が上限を超えた場合に報告する（docstring を除外した行数で計算）
- `max-file-length`: ファイルの行数が上限を超えた場合に報告する
- `no-testing-test-code`: `tests/` 配下のコードに対するテストコードの作成を禁止する

### ルールオプション設定

以下のルールはオプションによって動作をカスタマイズできる。

| ルール ID | オプション | デフォルト値 |
|-----------|-----------|-------------|
| `no-third-party-import` | `allow-dirs` | `[]`（許可ディレクトリなし） |
| `no-cross-package-import` | `allow-dirs` | `[]`（許可ディレクトリなし） |
| `max-method-length` | `max-lines` | `50` |
| `max-method-length` | `max-test-lines` | `100` |
| `max-class-length` | `max-lines` | `200` |
| `max-class-length` | `max-test-lines` | `400` |
| `max-file-length` | `max-lines` | `300` |
| `max-file-length` | `max-test-lines` | `500` |

- `allow-dirs` にはディレクトリパスを複数指定できる（`src/paladin/foundation/` や `tests/fake/` 等）
- `max-lines` と `max-test-lines` は整数値で指定する
- テストファイル（`tests/` 配下のファイル）には `max-test-lines` の値が適用される

### ルールメタ情報

- 各ルールはメタ情報として以下の項目を持つ
    - ルール ID
    - ルール名
    - 概要
    - 意図
    - 診断ガイダンス
    - 改善提案

### 違反情報

- 各違反は以下の項目を含む
    - ファイルパス
    - 行番号
    - 列番号
    - ルール ID
    - ルール名
    - 診断メッセージ
    - 理由
    - 改善提案

## 品質要件

### 拡張性

新しいルールを追加する際に、既存のルールや実行基盤のコードを変更せずに追加できること。

### テスト容易性

ルールの検査ロジックはファイルシステムや外部依存なしに単体テストできること。
テスト用のフェイク実装を外部から提供できること。

### 不変性

ソースファイル・違反・メタ情報などの値オブジェクトはすべて不変であること。
生成後に状態が変更されないことを保証すること。

### 検出の安定性と再現性

同一の入力ファイル・同一のルール集合・同一の設定に対して、常に同一の違反結果を返すこと。
検出結果が実行ごとに変動しないこと。

---

## 前提条件

### 技術基盤

- Python 3.11 以上
- Python 標準ライブラリの AST モジュールを利用してソースコードを解析する

### 解析対象の前提

- Python の `src` レイアウト（`src/<project>/<module>/`）を想定したパッケージ解決を行う
- `tests/` 配下のファイルをテストファイルとして扱う
