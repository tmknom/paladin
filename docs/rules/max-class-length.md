# max-class-length

## メタ情報

| 項目 | 値 |
|------|-----|
| ルールID | max-class-length |
| 対象 | 単一ファイル |

## 概要

単一クラスの行数が設定された上限を超えた場合に違反を検出します。プロダクションコードとテストコードで異なる上限を設定できます。

## 背景と意図

クラスの行数は、設計の健全性を示すシンプルかつ信頼性の高い指標です。100行のクラスと1万行のクラスでは、変更容易性もテスタビリティも根本的に異なります。

行数が増加することには、以下のような問題が伴います。

- **責務の肥大化** — 1つのクラスが多くの責務を持つほど、行数は増える。行数の超過は単一責任原則が破られているサインである
- **変更容易性の低下** — クラスが大きいほど、変更の影響範囲が広がり、修正コストが増大する
- **テスタビリティの低下** — 行数が多いクラスは依存関係が複雑になりやすく、テストの記述と維持が困難になる
- **認知負荷の増大** — 開発者がクラス全体を把握するために要するコストが、行数に比例して増加する

短いクラスが常に正しいわけではありませんが、長いクラスは間違いなく問題の兆候です。行数に上限を設けることで、設計上の問題を早期に発見する機会を得られます。

テストコードはテストケースが増えるほど行数が増える傾向にあるため、プロダクションコードより緩やかな上限を設定します。

## 診断メッセージ

| フィールド | 内容 |
|-----------|------|
| message | クラス `{class_name}` は `{length}` 行です。上限は `{limit}` 行です |
| reason | クラスが長すぎることは、責務の肥大化や設計上の問題を示す兆候です |
| suggestion | クラスの責務を分割し、複数のクラスに分離することを検討してください |

## 検出パターン

### 違反コード

```python
# src/paladin/check/orchestrator.py（上限200行に対して201行以上の場合）
# クラスの行数が設定された上限を超えている
class CheckOrchestrator:  # 違反: 250行のクラス
    def __init__(self, ...) -> None:
        ...

    def orchestrate(self, context: CheckContext) -> CheckReport:
        ...

    # （以降、行数が上限を超えて続く）
    ...
# クラス終端（250行目）
```

### 準拠コード

```python
# src/paladin/check/orchestrator.py（200行以内）
# クラスの行数が上限内に収まっている
class CheckOrchestrator:  # 準拠: 150行のクラス
    def __init__(self, ...) -> None:
        ...

    def orchestrate(self, context: CheckContext) -> CheckReport:
        ...
```

```python
# tests/unit/test_check/test_orchestrator.py（上限400行に対して400行以内）
# テストファイルは max-test-lines の上限が適用される
class TestCheckOrchestrator:  # 準拠: 380行のテストクラス
    def test_orchestrate_returns_report(self) -> None:
        ...

    def test_orchestrate_with_violations(self) -> None:
        ...
```

## 検出の補足

### 検出ロジック

1. 対象ファイルの AST を走査し、すべての `ClassDef` ノードを取得する
2. 各クラスの行数を算出する（「行数のカウント方法」参照）
3. 対象ファイルがテストファイルかどうかを判定する（「テストファイルの判定」参照）
4. テストファイルであれば `max-test-lines` の値を上限として使用し、それ以外は `max-lines` の値を使用する
5. クラスの行数が上限を超えた場合に違反を報告する

### 行数のカウント方法

`class` キーワードの行（`ClassDef.lineno`）からクラス本体の最終行（`ClassDef.body[-1]` の `end_lineno`）までの全行をカウントします。計算式は `end_lineno - lineno + 1` です。

空行・コメント行・docstring を含む、テキストとしての行数を基準とします。コードの意味的な行数ではなく、テキストとしての行数を基準とすることで、検出ロジックをシンプルに保ちます。

デコレータ（`@dataclass` 等）はクラス定義の外側に位置するため、行数に含めません。

### テストファイルの判定

ファイルのパスが `tests/` ディレクトリ配下にある場合、そのファイルをテストファイルとして扱います。具体的には、`pyproject.toml` からの相対パスが `tests/` で始まるファイルが対象です。

### ネストされたクラスの扱い

ネストされたクラスは独立した `ClassDef` ノードとして個別に検査します。外側のクラスの行数にはネストされたクラスの行数もテキスト行として含まれます。ネストされたクラス自体も、それ単体の行数が上限を超えた場合に違反を報告します。

### 報告の粒度

1ファイル内の各クラスを個別にチェックします。上限を超えたクラスごとに1件の違反を報告します。違反の行番号は `class` キーワードのある行とします。

### 設定ファイル

このルールは `max-lines` と `max-test-lines` パラメータで行数の上限を指定します。

```toml
[tool.paladin.rule."max-class-length"]
max-lines = 200
max-test-lines = 400
```

| パラメータ | 説明 | デフォルト値 |
|-----------|------|------------|
| `max-lines` | プロダクションコードのクラス行数上限 | `200` |
| `max-test-lines` | テストコードのクラス行数上限 | `400` |

`max-lines` と `max-test-lines` はそれぞれ独立して設定できます。未指定の場合はデフォルト値が使用されます。

ディレクトリ別に上限を変えたい場合は `[[tool.paladin.overrides]]` を使用します。

```toml
[[tool.paladin.overrides]]
files = ["scripts/**"]

[tool.paladin.overrides.rules]
max-class-length = false
```

### max-file-length との関係

`max-file-length` はファイル全体の行数を制限し、`max-class-length` はファイル内の個々のクラスの行数を制限します。両ルールは補完的な関係にあります。

- `max-file-length` は「ファイルに詰め込みすぎていないか」を検出する
- `max-class-length` は「個々のクラスが肥大化していないか」を検出する

1つの短いファイルに1つの巨大なクラスがある場合は `max-class-length` のみが違反を報告し、小さなクラスが大量にある長大なファイルは `max-file-length` のみが違反を報告します。

## 既存ツールとの関係

Pylint はクラスの属性数（`R0902`）やパブリックメソッド数（`R0904`）の上限を設定できますが、クラスの行数そのものを制限するルールは持ちません。Ruff も Pylint カテゴリの移植として `PLR0902` / `PLR0904` を提供していますが、クラスの行数制限は含まれません。

Paladin では、行数という「直接的で偽りのない指標」でクラスの肥大化を検出します。メソッド数が少なくても各メソッドが長ければクラスは膨れ上がるため、行数はメソッド数や属性数では捕捉できない問題も検出できる包括的な指標です。加えて、テストコードとプロダクションコードで異なる上限を単一の設定で区別でき、message/reason/suggestion の3フィールドで責務分割への具体的な修正行動を誘導します。
