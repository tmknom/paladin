# max-method-length

## メタ情報

| 項目 | 値 |
|------|-----|
| ルールID | max-method-length |
| 対象 | 単一ファイル |

## 概要

単一メソッド/関数の行数が設定された上限を超えた場合に違反を検出します。プロダクションコードとテストコードで異なる上限を設定できます。

## 背景と意図

メソッド/関数の行数は、設計の健全性を示すシンプルかつ信頼性の高い指標です。10行のメソッドと1000行のメソッドでは、変更容易性もテスタビリティも根本的に異なります。

行数が増加することには、以下のような問題が伴います。

- **責務の肥大化** — 1つのメソッド/関数が多くの処理を持つほど、行数は増える。行数の超過は単一責任原則が破られているサインである
- **変更容易性の低下** — メソッド/関数が大きいほど、変更の影響範囲が広がり、修正コストが増大する
- **テスタビリティの低下** — 行数が多いメソッド/関数は分岐やパスが複雑になりやすく、テストの記述と維持が困難になる
- **認知負荷の増大** — 開発者がメソッド/関数全体を把握するために要するコストが、行数に比例して増加する

短いメソッドが常に正しいわけではありませんが、長いメソッドは間違いなく問題の兆候です。行数に上限を設けることで、設計上の問題を早期に発見する機会を得られます。

テストコードはテストケースのセットアップやアサーションが増えるほど行数が増える傾向にあるため、プロダクションコードより緩やかな上限を設定します。

## 診断メッセージ

| フィールド | 内容 |
|-----------|------|
| message | メソッド/関数 `{function_name}` は `{length}` 行です。上限は `{limit}` 行です |
| reason | メソッド/関数が長すぎることは、責務の肥大化や設計上の問題を示す兆候です |
| suggestion | メソッド/関数の処理を分割し、複数のプライベートメソッドや関数に分離することを検討してください |

## 検出パターン

### 違反コード

```python
# src/paladin/check/orchestrator.py（上限50行に対して51行以上の場合）
# メソッドの行数が設定された上限を超えている
class CheckOrchestrator:
    def orchestrate(self, context: CheckContext) -> CheckReport:  # 違反: 80行のメソッド
        ...

        # （以降、行数が上限を超えて続く）
        ...
    # メソッド終端（80行目）
```

### 準拠コード

```python
# src/paladin/check/orchestrator.py（50行以内）
# メソッドの行数が上限内に収まっている
class CheckOrchestrator:
    def orchestrate(self, context: CheckContext) -> CheckReport:  # 準拠: 40行のメソッド
        ...
```

```python
# tests/unit/test_check/test_orchestrator.py（上限100行に対して100行以内）
# テストファイルは max-test-lines の上限が適用される
class TestCheckOrchestrator:
    def test_orchestrate_returns_report(self) -> None:  # 準拠: 90行のテストメソッド
        ...
```

## 検出の補足

### 検出ロジック

1. 対象ファイルの AST を走査し、すべての `FunctionDef` および `AsyncFunctionDef` ノードを取得する
2. 各メソッド/関数の行数を算出する（「行数のカウント方法」参照）
3. 対象ファイルがテストファイルかどうかを判定する（「テストファイルの判定」参照）
4. テストファイルであれば `max-test-lines` の値を上限として使用し、それ以外は `max-lines` の値を使用する
5. メソッド/関数の行数が上限を超えた場合に違反を報告する

### 行数のカウント方法

`def` キーワードの行（`FunctionDef.lineno`）からメソッド/関数本体の最終行（`FunctionDef.body[-1]` の `end_lineno`）までの全行をカウントします。計算式は `end_lineno - lineno + 1` です。

空行・コメント行・docstring を含む、テキストとしての行数を基準とします。コードの意味的な行数ではなく、テキストとしての行数を基準とすることで、検出ロジックをシンプルに保ちます。

デコレータ（`@classmethod` 等）はメソッド/関数定義の外側に位置するため、行数に含めません。

### テストファイルの判定

ファイルのパスが `tests/` ディレクトリ配下にある場合、そのファイルをテストファイルとして扱います。具体的には、`pyproject.toml` からの相対パスが `tests/` で始まるファイルが対象です。

### ネストされた関数の扱い

ネストされた関数（inner function）は独立した `FunctionDef` ノードとして個別に検査します。外側のメソッド/関数の行数にはネストされた関数の行数もテキスト行として含まれます。ネストされた関数自体も、それ単体の行数が上限を超えた場合に違反を報告します。

### 報告の粒度

1ファイル内の各メソッド/関数を個別にチェックします。上限を超えたメソッド/関数ごとに1件の違反を報告します。違反の行番号は `def` キーワードのある行とします。

### 設定ファイル

このルールは `max-lines` と `max-test-lines` パラメータで行数の上限を指定します。

```toml
[tool.paladin.rule."max-method-length"]
max-lines = 50
max-test-lines = 100
```

| パラメータ | 説明 | デフォルト値 |
|-----------|------|------------|
| `max-lines` | プロダクションコードのメソッド/関数行数上限 | `50` |
| `max-test-lines` | テストコードのメソッド/関数行数上限 | `100` |

`max-lines` と `max-test-lines` はそれぞれ独立して設定できます。未指定の場合はデフォルト値が使用されます。

ディレクトリ別に上限を変えたい場合は `[[tool.paladin.overrides]]` を使用します。

```toml
[[tool.paladin.overrides]]
files = ["scripts/**"]

[tool.paladin.overrides.rules]
max-method-length = false
```

### max-class-length / max-file-length との関係

`max-file-length`・`max-class-length`・`max-method-length` は、それぞれファイル・クラス・メソッド/関数という粒度の異なるレベルで行数を制限する補完的なルールです。

- `max-file-length` は「ファイルに詰め込みすぎていないか」を検出する
- `max-class-length` は「個々のクラスが肥大化していないか」を検出する
- `max-method-length` は「個々のメソッド/関数が肥大化していないか」を検出する

例えば、クラス全体は小さくてもメソッドが長い場合は `max-method-length` のみが違反を報告し、メソッドは短いがクラスに多数のメソッドが詰め込まれている場合は `max-class-length` のみが違反を報告します。3つのルールを組み合わせることで、異なる粒度の設計問題を網羅的に検出できます。

## 既存ツールとの関係

Ruff は `PLR0915`（`too-many-statements`）で関数内のステートメント数を制限できますが、行数そのものを制限するルールは持ちません。Pylint も `R0915` として同様のステートメント数制限を提供していますが、行数制限ではありません。

Paladin では、行数という「直接的で偽りのない指標」でメソッド/関数の肥大化を検出します。ステートメント数が少なくても各ステートメントが長い式や複雑な条件式で構成されていれば、メソッドは膨れ上がります。行数はステートメント数では捕捉できない問題も検出できる包括的な指標です。加えて、テストコードとプロダクションコードで異なる上限を単一の設定で区別でき、message/reason/suggestion の3フィールドで責務分割への具体的な修正行動を誘導します。
