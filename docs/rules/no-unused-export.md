# no-unused-export

## メタ情報

| 項目 | 値 |
|------|-----|
| ルールID | no-unused-export |
| 対象 | 複数ファイル |

## 概要

`__init__.py` の `__all__` に定義したシンボルが、プロダクションコードの別パッケージから実際に利用されていない場合に違反を検出するルールです。利用されていない公開シンボルは不必要な公開であり、削除することでパッケージの公開インタフェースを最小化できます。

## 背景と意図

`__init__.py` の `__all__` はパッケージの公開インタフェースを明示的に定義するための仕組みです。公開したシンボルはそのパッケージの利用者に向けた契約であり、以下の責任が生じます。

- シンボルの後方互換性を維持する責任（削除・リネームが難しくなる）
- ドキュメント・テスト・型情報を含む維持コストの負担
- パッケージ境界の明確さの低下（利用者がどのシンボルを使えばよいか判断しにくくなる）

別パッケージから利用されていないシンボルが `__all__` に残り続ける状況は、以下の問題を引き起こします。

- YAGNI 原則への違反: 実際に必要とされていないにもかかわらず、将来の利用を想定して公開している状態
- 暗黙のメンテナンス義務: 誰も使っていないシンボルでも、公開している以上は後方互換性の観点から変更しにくい
- パッケージの肥大化: 公開シンボルが増えるほど、パッケージのインタフェースが複雑になり、利用者の理解コストが上がる

`__all__` に列挙するシンボルを実際に利用されているものだけに絞り込むことで、公開インタフェースを最小化し、内部実装の自由度とメンテナンス性を高めます。

## 診断メッセージ

| フィールド | 内容 |
|-----------|------|
| message | `__all__` のシンボル `{name}` はどの別パッケージからも利用されていない |
| reason | 利用されていないシンボルを公開し続けると、不必要な後方互換義務が生じる |
| suggestion | `{name}` を `__all__` から削除してください |

## 検出パターン

### 違反コード

```python
# paladin/check/__init__.py
from paladin.check.orchestrator import CheckOrchestrator
from paladin.check.context import CheckContext

__all__ = [
    "CheckOrchestrator",  # 違反: 別パッケージからインポートされていない
    "CheckContext",
]
```

```python
# paladin/rule/__init__.py  ← 実際にチェックを利用しているパッケージ
from paladin.check import CheckContext  # CheckOrchestrator は使っていない

class SomeOrchestrator:
    def run(self, context: CheckContext) -> None:
        ...
```

### 準拠コード

```python
# paladin/check/__init__.py
from paladin.check.orchestrator import CheckOrchestrator
from paladin.check.context import CheckContext

__all__ = [
    "CheckOrchestrator",  # 準拠: 別パッケージから実際に利用されている
    "CheckContext",       # 準拠: 別パッケージから実際に利用されている
]
```

```python
# paladin/rule/__init__.py  ← 実際に両方を使っているパッケージ
from paladin.check import CheckOrchestrator, CheckContext
```

```python
# paladin/check/__init__.py — CheckOrchestrator を公開しない設計に変更
from paladin.check.context import CheckContext

__all__ = [
    "CheckContext",  # 準拠: 実際に利用されているシンボルだけを公開
]
```

## 検出の補足

### 対象ファイル

このルールは `__init__.py` のみを対象とします。通常のモジュールファイル（`__init__.py` 以外）の `__all__` は対象外です。

### テストコードの扱い

export 元がプロダクションコードかテストコードかによって、利用のカウント対象が異なります。

- **プロダクションコードの `__all__`**: プロダクションコード（`tests/` 以外）からの利用のみカウントします。テストコードからのインポートは内部実装の検証目的であることが多く、公開インタフェースとしての利用とは性質が異なるため、カウントしません。
- **テストコードの `__all__`**: テストコードからの利用もカウントします。テストパッケージの公開シンボルはテスト間でのみ参照されるものであり、テストコードからの利用が「利用されている」状態を意味します。

### 検出ロジック

- 各 `__init__.py` の `__all__` に列挙されているシンボルを収集する
- プロダクションコード全体を走査し、`from <package> import <name>` または `import <package>` 後の `<package>.<name>` 参照の形式で利用されているシンボルを収集する
- `__all__` のシンボルのうち、上記参照に含まれないものを違反として報告する
- 同一パッケージ内（`__init__.py` 自身が属するパッケージ）からの利用は「別パッケージからの利用」とは見なさない。他パッケージからの参照が1件以上あれば準拠

## 既存ツールとの関係

Ruff には `__all__` に列挙されたシンボルが別パッケージから参照されているかを検出するルールはありません。`F401`（未使用インポート）や `PLC0414`（再エクスポートの明示化）は別の観点のルールであり、このルールが検出する「公開したが利用されていない」状況は対象外です。

Pylint にも同等のルールは存在しません。パッケージの公開インタフェースが実際に利用されているかという設計上の観点は、既存のリンターでは扱われていない領域であり、Paladin で独自に扱います。
