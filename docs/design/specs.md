# 仕様ドキュメント設計

## 目的

`docs/specs/` ディレクトリ配下で管理する「仕様ドキュメント」の目的は、LLMに必要なコンテキストを明示的に提供することにある。
セッションをまたいでも、設計思想に沿ったコード変更を行えるようにする。

対象読者はLLMおよびこのプロジェクトを保守するエンジニアの両方。

## 設計思想

LLMにはセッションごとにメモリがリセットされるという特性がある。
また、ファイル探索ツール（Explorer）でたまたま見つけた情報に引っ張られる傾向がある。
質の高い情報を明示的に提供することで、設計思想に沿ったコード変更を促す。
以上の前提から、次の方針を採用する。

- **ドキュメントを読んでから変更する**: LLMは `llms.txt` → `docs/specs/` の順で読み込み、その後コード変更を行う
- **意図的な冗長性をもたせる**: ファイルを見れば分かること（ファイル名とクラス名のマッピングなど）もあえて記載する。冗長に見えても、LLMに正確なコンテキストを与えるためには必要
- **設計の意図を明示する**: 「何があるか」だけでなく「なぜそう設計したか」を記述する。意図が伝わらないと、LLMが合理的に見える別の設計へ変更するリスクがある

### 冗長性の判断基準

「意図的な冗長性」には歯止めが必要である。次の3つの観点で判断する。

- **全体最適に寄与するか**: 複数モジュールにまたがる設計判断、依存関係のマッピング → 書く
- **誤った変更を防ぐか**: ガードレール・禁止事項・壊してはいけない不変条件 → 書く
- **局所最適にしか寄与しないか**: 1ファイル内で完結する実装詳細（コードを読めばわかる） → 書かない

## 要件定義書の役割

### 目的

「何を実現するか（What）」を技術実装から分離して記述する。
LLMが変更の妥当性を判断したり、「これはバグか仕様か」を判定したりするための参照先となる。

### 書くべきもの

- モジュールの目的と解決する課題
- 機能要件（「〜できる」形式で列挙）
- 品質要件（不変性・テスト容易性・拡張性など機能要件以外で満たすべき品質特性）
- 前提条件（技術基盤、利用シナリオ、外部との制約など）

### 書くべきでないもの

- クラス名・関数名・変数名などの実装詳細
- アーキテクチャパターンや設計判断（design.md の責務）
- 処理フローの技術的詳細（design.md の責務）

## 基本設計書の役割

### 目的

「どう実現するか（How）＋なぜそう設計したか（Why）」を記述する。
LLMが設計意図に沿った変更を行えるようにし、「してはいけないこと」を明示する。

### 書くべきもの

- コンポーネント構成とファイルマッピング（クラス名とファイル名の対応、パッケージ構成の全体像）
- 設計の意図とトレードオフ（採用した設計の理由、却下した代替案、制約の背景）
- ガードレール（壊してはいけない不変条件、混在させてはいけない責務）
- 外部依存（自身の制御下にない外部システム・サードパーティライブラリへの依存と対策）

### 書くべきでないもの

- 関数の詳細な実装やロジック（コードを読めばわかる）
- 変数名・パラメータ名の詳細
- 要件の列挙のみで設計判断の根拠がない記述（requirements.md の責務）
- モジュール外のコンポーネントに関する情報（呼び出し元、他モジュールの内部構造）
- テストや静的解析で検出可能な制約（公開 API の範囲、import 制約など）
- 実装の写経（リファクタリングで容易に陳腐化する情報）

### 設計判断の記述フォーマット

設計判断は以下の3要素で記述する。

- **設計の意図**: その設計で何を実現したいのか
- **なぜそう設計したか**: 理由と代替案を却下した根拠
- **トレードオフ**: その設計の利点と受け入れた欠点

## 基本設計書の情報管理方針

上記の設計思想を前提としつつ、基本設計書には以下の追加原則を適用する。

なお「冗長性の判断基準」における「全体最適に寄与するか」については、複数モジュールにまたがる情報はアーキテクチャ設計ドキュメント等が受け皿となり、基本設計書はモジュールスコープの原則に従って自モジュールに閉じる。

### 正確性と網羅性のトレードオフ

情報の正確性を最優先し、網羅性は犠牲にする。誤情報を書くなら何も書いてないほうがマシ。
運用上、正確性を維持できない情報は記載しない。

### モジュールスコープの原則

基本設計ドキュメントのスコープは自モジュールに閉じる。モジュール外の情報は書かない。

理由: 仕様ドキュメントの同期はモジュール単位で行う。モジュール外の情報は、そのモジュールを変更しても同期対象にならないため、陳腐化が避けられない。

例外: 自身の制御下にない外部依存（外部システム API、サードパーティライブラリ）は記載する。予期せぬタイミングで影響を受ける可能性があり、コストをかけて追従する価値がある。

### 実装詳細の記載基準

「コンポーネント構成」セクション以外では、実装詳細（クラス名・メソッド名・処理の呼び出し順序）を最小限にする。

テストや静的解析で検出可能な事柄は記載しない。コードやツールで表現できない情報（設計意図、トレードオフ、不変条件）にフォーカスする。

## 構成ファイル

各モジュールの仕様ドキュメントは、最低限以下の2ファイルで構成する。

| ファイル | 内容 |
|---------|------|
| `requirements.md` | 要件定義。モジュールの目的・機能要件・品質要件・前提条件 |
| `design.md` | 基本設計。コンポーネント構成・設計の意図・ガードレール・外部依存 |

これら2ファイルでは収まらない情報（例：詳細なCLIインターフェース仕様、APIリファレンス）が存在する場合は、別ファイルを追加してよい。

## 運用フロー

### コード変更前

LLMは以下の順で読み込む:

```
llms.txt（ドキュメントインデックス）
    ↓
docs/specs/（対象モジュールの仕様書）
    ↓
コード変更
```

### コード変更後

コード変更後は必ず、仕様ドキュメントの同期タスクを追加する（`CLAUDE.md` の Plan Mode Constraints に記載済み）。

同期が漏れると、仕様ドキュメントが「高品質な誤情報」をLLMに与えることになる。
実装との乖離は有害であり、同期は必須。

## ガードレール

| ルール | 理由 |
|-------|------|
| 仕様ドキュメントと実装が乖離した場合は必ず同期する | 古い仕様ドキュメントは誤情報化し、LLMを誤った方向へ誘導する |
| 仕様ドキュメントを削除・大幅に簡略化してはならない | 冗長に見えても、LLMへの情報提供という意図がある |
