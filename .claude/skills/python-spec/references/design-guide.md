# 基本設計書ガイドライン

Python モジュールの基本設計書（design.md）作成・保守のためのガイドラインです。
要件定義書で定義された機能を「どう実現するか」「なぜそう設計したか」を、経緯を知らないエンジニアやLLMが理解できる形で記述します。

## 方針

### 目的

基本設計書は「どう実現するか（How）＋なぜそう設計したか（Why）」を記述するドキュメントである。

対象読者はLLMおよび保守エンジニア。LLMにとっては、設計意図に沿った変更を行うためのナビゲーションマップとして機能する。具体的には以下の問いに答える。

- どこを変更すればよいか
- なぜこう設計されているか
- してはいけないことは何か

### 書くべきもの

- コンポーネント構成とファイルマッピング（クラス名とファイル名の対応、パッケージ構成の全体像）
- 設計の意図とトレードオフ（採用した設計の理由、却下した代替案、制約の背景）
- ガードレール（壊してはいけない不変条件、混在させてはいけない責務）
- 外部依存（自身の制御下にない外部システム・サードパーティライブラリへの依存と、その影響の隔離方法）

### 書くべきでないもの

- 関数の詳細な実装やロジック（コードを読めばわかる）
- 変数名・パラメータ名の詳細
- 要件の列挙のみで設計判断の根拠がない記述（requirements.md の責務）
- モジュール外のコンポーネントに関する情報（呼び出し元、他モジュールの内部構造など）
- テストや静的解析で検出可能な制約（公開 API の範囲、import 制約など）
- 実装の写経（リファクタリングで容易に陳腐化する情報）
- アーキテクチャ全体設計で既に説明されている内容（共通パターンの説明、レイヤー構造、横断的設計方針）

### 情報の取捨選択基準

基本設計ドキュメントに掲載する情報は、以下の原則に基づいて厳選する。

**正確性 > 網羅性**

誤情報を書くなら何も書いてないほうがマシ。重要な情報と全体像の把握に役立つ情報を最優先し、局所的な情報は書かない。運用上、正確性を維持できない情報は記載しない。

**モジュールスコープの厳守**

基本設計ドキュメントのスコープは自モジュールに閉じる。モジュール外の情報は書かない。

理由: 仕様ドキュメントの同期はモジュール単位で行う。モジュール外の情報は、そのモジュールを変更しても同期対象にならないため、陳腐化が避けられない。

例外: 自身の制御下にない外部依存（外部システム API、サードパーティライブラリ）は記載する。予期せぬタイミングで影響を受ける可能性があり、コストをかけて追従する価値がある。

**コードやツールで表現できない情報にフォーカスする**

テストや静的解析で検出可能な事柄は書かない。コードを読めばわかる実装詳細は書かない。コードやツールで表現できない情報（設計意図、トレードオフ、不変条件）にフォーカスする。

**「コンポーネント構成」セクション以外では実装詳細を最小限にする**

クラス名・メソッド名への言及は、コンポーネント構成セクションでは必要だが、他セクションでは最小限にとどめる。リファクタリングで壊れる情報は書くべきでない。

**アーキテクチャ全体設計との重複を避ける**

プロジェクトにはアーキテクチャ全体設計のドキュメントが存在し、コード変更時に毎回コンテキストに読み込まれる。基本設計ドキュメントでは、アーキテクチャ全体設計に記載済みの情報（レイヤー構造、共通パターンの説明、横断的設計方針など）を繰り返さない。

基本設計ドキュメントがフォーカスすべきは、アーキテクチャ全体設計のスコープ外にある、モジュール固有の情報である。

- 書くべき: 当モジュールが採用したパターンとその適用箇所（表形式での列挙）、モジュール固有の設計判断・制約・処理フロー
- 書くべきでない: パターンそのものの説明、レイヤー構造の再説明、全体設計で既に述べられている設計方針の繰り返し

### 陳腐化への注意

仕様ドキュメントと実装の乖離は有害である。以下を意識する。

- 自モジュールの変更後は必ず design.md を同期する
- モジュールの責務が別パッケージへ移動した場合は、該当する記述を削除する（または移動先の設計ドキュメントへ移す）

### 設計判断の記述フォーマット

設計判断は以下の3要素で記述する。

- **設計の意図**: その設計で何を実現したいのか
- **なぜそう設計したか**: 理由と代替案を却下した根拠
- **トレードオフ**: その設計の利点と受け入れた欠点

## 推奨構成

モジュールの性質に応じて、セクションの追加・削除は可能です。

```markdown
# [モジュール名] 基本設計

[要件定義](./requirements.md) に基づいた基本設計を説明します。

## アーキテクチャパターン

## コンポーネント構成

### 主要コンポーネント

### ファイルレイアウト

## 処理フロー

## 固有の設計判断

### [設計判断のタイトル]

## 制約と注意点

### 副作用

## 外部依存

### 外部システムへの依存

### サードパーティライブラリへの依存

## 関連ドキュメント
```

## 各セクションの書き方

### アーキテクチャパターン

当モジュールが採用しているアーキテクチャパターンを表形式で列挙する。

- パターン名・適用箇所・目的を含める
- プロジェクト共通の設計思想への参照を冒頭に置く
- パターンそのものの説明はアーキテクチャ全体設計の責務であるため、当モジュールでの適用に絞る

### コンポーネント構成

#### 主要コンポーネント

クラス名・役割の対応を表形式で列挙する。LLMが「どのクラスがどの責務を持つか」を素早く把握するためのセクション。

#### ファイルレイアウト

プロダクションコードとテストコードのディレクトリツリーを記載する。

- bash コードブロックでディレクトリツリーを表現する
- ファイル名の横にコメントで主要なクラス名を記載する
- テストコードは、ユニットテストだけでなく関係の深いインテグレーションテストや E2E テストも記載する

### 処理フロー

データの流れと処理ステップをハイレベルに記述する。

- 番号付きリストでハイレベルな処理ステップ（3〜5ステップ程度）を列挙する
- 要件定義が変わったら変わるレベルの抽象度で記述する
- 実装が少し変わっただけで陳腐化する詳細（サブステップ、内部の呼び出し順序）は書かない
- 各ステップで関与する主要コンポーネント名への言及は許容する
- 複雑なフローは Mermaid 図で視覚化する

### 固有の設計判断

design.md の中核的価値を持つセクション。モジュール固有の設計判断を記述する。

以下の基準で記載対象を選ぶ。

- 書くべき: 複数コンポーネントにまたがる設計判断、誤った変更を防ぐガードレール的判断、コードから読み取りにくい設計意図
- 書くべきでない: 1ファイル内で完結する実装詳細、クラスが増えるたびに機械的に追加する説明

自モジュールの責務に閉じた設計判断のみ記述する。他モジュールの設計判断や依存構造は記載しない。

各設計判断は以下の構造で記述する。

1. **設計の意図**: その設計で何を実現したいのかを1〜2文で述べる
2. **なぜそう設計したか**: 代替案との比較や制約条件を含め、選択の理由を説明する
3. **トレードオフ**: その設計の受け入れた欠点と、将来的に問題になりうるケースを述べる

### 制約と注意点

コードやテストで検出できない不変条件、混在させてはいけない責務を記述する。

- 書くべき: データの順序保証・一貫性要件など、テストで検出しにくい不変条件
- 書くべきでない: 公開 API の範囲（`__init__.py` を読めばわかる）、import 制約（静的解析で検出可能）

#### 副作用

モジュール内で副作用が発生する箇所を明記する。副作用には、ファイル I/O・ネットワークアクセス・外部プロセス実行などが該当する。

副作用の隔離方法（Protocol による抽象化など）がある場合は簡潔に言及する。詳細な設計判断は「固有の設計判断」セクションに書く。

副作用がない場合は「副作用はない」と明記する。

### 外部依存

自身の制御下にない外部システムやサードパーティライブラリへの依存を記述する。

自リポジトリ内の他パッケージへの依存は記載しない。import 文を読めばわかるため。

#### 外部システムへの依存

外部 API・外部サービスなど、自身の制御下にないシステムとのインタラクションを記述する。接続先・用途・影響の隔離方法を含める。

該当がない場合は「外部システムへの依存はない」と明記する。

#### サードパーティライブラリへの依存

Python 標準ライブラリ以外のサードパーティライブラリへの依存を表形式で列挙する。ライブラリ名と用途を含める。

該当がない場合は「サードパーティライブラリへの依存はない」と明記する。

## 表現のガイドライン

### 図の活用

構造や処理フローの視覚化には Mermaid を活用する。ファイルレイアウトは bash コードブロックでディレクトリツリーとして表現する。

### 記述の抽象度

| レベル | 記載すべき内容 | 例 |
|---|---|---|
| アーキテクチャ | 記載する | オニオンアーキテクチャ、Composition Root |
| 設計パターン | 記載する | Provider パターン、Protocol 抽象化 |
| 主要クラス名 | 記載する | Orchestrator、Formatter |
| 処理フロー | 記載する | ハイレベルな処理ステップ |
| 関数の実装 | 記載しない | ループの書き方、変数名 |

### コード例の使用

必要に応じてコード例を使用する。ただし型定義やインターフェースの提示に限り、実装の詳細は避ける。

## チェックリスト

### 基本要件

- [ ] 冒頭に要件定義書へのリンクがある
- [ ] 推奨構成に従っている（または合理的な理由で変更している）

### 内容品質

- [ ] 主要コンポーネントとファイルレイアウトが記載されている
- [ ] 処理フローが記述されている
- [ ] 固有の設計判断セクションで3要素（設計の意図・理由・トレードオフ）が説明されている
- [ ] 制約と注意点（ガードレール）が記述されている
- [ ] 制約と注意点に副作用サブセクションがある（副作用がない場合もその旨を明記）
- [ ] 外部依存が記載されている（外部システム・サードパーティライブラリとも、該当なしの場合もその旨を明記）

### 責務の逸脱チェック

- [ ] 要件の列挙のみで設計判断の根拠がない記述がない
- [ ] 関数の詳細な実装やロジックが含まれていない
- [ ] モジュール外のコンポーネントに関する情報が含まれていない
- [ ] テストや静的解析で検出可能な制約が含まれていない
- [ ] アーキテクチャ全体設計で説明済みの内容を繰り返していない

### 陳腐化チェック

- [ ] コンポーネント構成が実装と一致している
- [ ] ファイルレイアウトが実際のディレクトリ構造と一致している
