# 要件定義書ガイドライン

Python モジュールの要件定義書（requirements.md）作成・保守のためのガイドラインです。

## 方針

### 目的

要件定義書は「何を実現するか（What）」を技術実装から分離して記述するドキュメントである。

対象読者はLLMおよび保守エンジニア。LLMにとっては、変更の妥当性判断や「これはバグか仕様か」の判定基準として機能する。

### 書くべきもの

- モジュールの目的と解決する課題
- 機能要件（「〜できる」形式で列挙）
- 品質要件（不変性・テスト容易性・拡張性など機能要件以外で満たすべき品質特性）
- 前提条件（技術基盤、利用シナリオ、外部との制約など）
- ユーザー向け契約仕様（CLI フラグ、設定ファイル構文、コメントディレクティブ構文、環境変数名、出力フォーマット仕様）

### 書くべきでないもの

- 内部実装詳細（クラス名・関数名・変数名・アルゴリズム選択）
- アーキテクチャパターンや設計判断（design.md の責務）
- 処理フローの技術的詳細（design.md の責務）
- 曖昧な表現（「許容範囲内」「適切な」等）

### 契約仕様と内部実装詳細の区別

「技術実装から分離する」とは、外部から観測できない内部構造を排除することであり、ユーザー向けインターフェースを排除することではない。

**書くべきもの（契約仕様）**: ユーザーに公開されたインターフェース。変更するとユーザーのワークフローが壊れるもの。

| 具体例 | 判定 |
|--------|------|
| `--format json` CLI オプション | 契約仕様 → 書く |
| `[tool.paladin.per-file-ignores]` 設定セクション | 契約仕様 → 書く |
| `# paladin: ignore-file` コメント構文 | 契約仕様 → 書く |
| `EXAMPLE_LOG_LEVEL` 環境変数名 | 契約仕様 → 書く |
| `{"rules": [...]}` JSON 出力構造 | 契約仕様 → 書く |

**書くべきでないもの（内部実装詳細）**: 外部から観測できない内部構造。変更しても外部の動作は変わらないもの。

| 具体例 | 判定 |
|--------|------|
| `CheckOrchestrator` クラス名 | 内部実装詳細 → 書かない |
| `_collect_files()` 関数名 | 内部実装詳細 → 書かない |
| `pydantic-settings` ライブラリ使用（前提条件として書く場合は除く） | 内部実装詳細 → 書かない |
| AST ノードのトラバーサル方法 | 内部実装詳細 → 書かない |

## 推奨構成

モジュールの性質に応じて、セクションの追加・削除は可能です。

```markdown
# [モジュール名] 要件定義

## 概要

### 目的
### 解決する課題
### コンポーネントの概要

## 機能要件

### [機能グループ1]
### [機能グループ2]

## 品質要件

### [品質項目1]
### [品質項目2]

## 前提条件

### 技術基盤

## 関連ドキュメント
```

## 各セクションの書き方

### 概要セクション

#### 目的

モジュールが実現することを簡潔に記述する。

- 1〜2段落で簡潔に
- モジュールが提供する価値を明確に
- 技術詳細は不要

#### 解決する課題

現状の課題を箇条書きで列挙する。

- 3〜5項目程度
- 具体的に記述

#### コンポーネントの概要

モジュールの全体像を1〜2段落で記述する。

- 入力 → 処理 → 出力の流れを明確に
- 提供するコンポーネントの概要を列挙
- 技術詳細は避ける

### 機能要件セクション

モジュールが提供する機能を列挙する。

- 「〜できる」形式で記述する
- 機能グループごとにサブセクションで分類する
- 実装方法ではなく機能の内容を記述する
- ユーザー向け契約仕様（CLI フラグ・設定構文・コメントディレクティブ等）は具体的に記述する

### 品質要件セクション

機能要件以外で満たすべき品質特性を記述する。

- 具体的な数値目標がある場合は記載する
- ない場合は定性的な記述でよい
- 実現方法ではなく達成すべき品質を記述する

### 前提条件セクション

モジュールが動作するために必要な前提を記述する。

- 大前提となる技術スタックは記載してよい
- 実装詳細レベルは不要

## チェックリスト

### 基本要件

- [ ] 推奨構成に従っている（または合理的な理由で変更している）
- [ ] 対象読者が理解できる表現になっている

### 内容品質

- [ ] モジュールの目的と解決する課題が明確に記述されている
- [ ] 機能要件が「何ができるか」で記述されている（「どう実装するか」ではない）
- [ ] 曖昧な表現（「許容範囲内」等）が含まれていない

### 責務の逸脱チェック

- [ ] 内部実装詳細（クラス名・関数名・アルゴリズム）が含まれていない
- [ ] アーキテクチャパターンや設計判断が含まれていない
- [ ] 処理フローの技術的詳細が含まれていない

ユーザー向け契約仕様（CLI フラグ・設定構文・コメントディレクティブ）は要件の一部であり、含めてよい。

### 関連ドキュメント

- [ ] 基本設計書へのリンクが設定されている
